Development History

Most recent updates first. Each entry summarizes system state, key events, and what was learned.

June 2026
System
v0.3: Master WBS Restructure, Data Flow Optimization & UX Overhaul
What changed

Major restructuring of the Master WBS Tasks database, with schema simplified and field organization redesigned to reduce redundancy and improve cross-project comparability. Data flow between Project WBS → Master WBS → Work Sessions was streamlined to eliminate intermediate steps and reduce sync latency. Interface optimizations across the Focal UI improved usability: cleaner layouts, reduced clicks for common actions, and more consistent visual hierarchy. Data entry was made more flexible and consistent through field defaults, option normalization, and validation rules applied across all project WBS databases to prevent silent mismatches during sync.

Why it was done

Daily use revealed friction points that weren't visible during initial design: fields that felt logical in schema design were confusing in practice; data entry inconsistencies across projects were producing noisy master views; and the interface had accumulated small inefficiencies that compounded over repeated use. This iteration was user-experience driven, not feature driven.

Key lessons from this cycle

Real use surfaces problems that design sessions cannot anticipate. The gap between "what seemed right in design" and "what works in daily practice" is a critical feedback loop that requires the human to be the user, not just the designer. AI can implement changes quickly once the friction is identified, but the identification itself requires lived experience with the system.

Implications for the portfolio

This iteration adds weight to the "real stakes" argument in the Case section, since the system is refined through actual use, not simulated testing. The UX-driven refactor also surfaces a new competency dimension: the ability to distinguish "this is a design problem" from "this is an implementation bug" when something feels wrong in daily use.

May 2026
System
v0.2: Modular Architecture Refactor
What changed

Monolithic notion_wbs_sync.py (~3,100 lines) refactored into a proper Python package: focal/app.py, config.py, notion_client.py, sync_engine.py, tasks.py, with route blueprints and HTML templates. Entry point moved to main.py. _MANIFEST.md Tier System introduced for context management.

Why it was done

The monolith became too large for efficient AI collaboration. Locating the right section required too much context. The refactor reduced per-session cognitive load and made the codebase easier to reason about for both human and AI.

Key lesson

The refactor wasn't triggered by a feature failure. The system worked. It was triggered by collaboration friction. This is an important data point: in human–AI projects, architecture quality directly affects collaboration quality, not just system quality.

Early 2026
System
v0.1: First Automation (Monolithic Script)
What was built

Single Python/Flask script that syncs per-project Notion WBS databases into a Master WBS Tasks database. Flask UI added for sync triggering and Quick Add. Work Sessions auto-creation added. Core sync rules established: one-directional, idempotent, status only set on CREATE.

Key design decision

One-directional sync (Project WBS → Master only). This was not obvious initially. The first implementation was bidirectional and immediately caused a status overwrite bug. The fix revealed a core architectural principle.
