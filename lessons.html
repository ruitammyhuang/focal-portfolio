Lessons Learned

What Actually Happened: Candid Lessons

Not curated successes — a real account of what worked, what failed, and what was surprising. Organized so educators can use these directly in teaching without needing to re-derive them from scratch.

What Works

Patterns That Consistently Produced Good Outcomes

Replicable practices that are worth teaching explicitly.

✓ Works Well
Treating the AI as a persistent collaborator, not a one-shot query tool
The highest-quality outputs came from conversations that built context over multiple exchanges — where the AI had learned (within a session) what the constraints were, what had already been tried, and what the human actually cared about. Sessions that opened with "do this task" produced worse results than sessions that opened with context, discussed approach, then implemented.
Teach students: design your sessions, not just your prompts. The conversation arc matters as much as any individual message.
✓ Works Well
Encoding domain knowledge in skills files rather than restating it each session
The most valuable investment in the Focal project was building skills files that encoded reusable domain knowledge — database IDs, sync rules, hub page patterns, config schema. These files function simultaneously as documentation (human-readable) and AI instruction (machine-actionable). Once written, they eliminated a class of re-derivation errors entirely.
Treat shared domain knowledge as an artifact to build, not just information to repeat. The skill files are a form of institutional memory for the human–AI team.
✓ Works Well
Letting the AI surface options before the human commits to an approach
When the human asked for "option A or option B?" the conversation was constrained. When the human asked "what are the main approaches here and what are the tradeoffs?" the AI often surfaced an option C the human hadn't considered — sometimes the best one. This required the human to resist the urge to jump straight to implementation.
Teach: delay commitment to approach until after the AI has surfaced the option space. "What are my options?" is often a better first prompt than "do X".
✓ Works Well
Using the AI to explain its own code back
Asking the AI to "explain what this function does in plain English" after generating it served two purposes: (1) it built the human's understanding of the implementation, and (2) it sometimes revealed that the AI's explanation didn't match its code — catching a bug before deployment. This became a standard verification step.
Make AI self-explanation a standard part of the session protocol. If the AI can't explain what it built in terms that match your requirements, something is wrong.
Pitfalls to Avoid

Patterns That Produced Problems

Things to avoid — and what to do instead.

✗ Pitfall
Assuming AI-generated code is correct because it looks reasonable
AI-generated code tends to look correct — it follows naming conventions, handles the obvious case, and is well-structured. This makes errors harder to spot than in hastily written code. The Focal status overwrite bug persisted for multiple sessions because the code was clean and the error was in the logic, not the syntax.
Never evaluate AI code by how it looks. Always evaluate it by whether it does what the requirements say it should do, in all the cases you care about.
✗ Pitfall
Under-specifying requirements and over-attributing failures to the AI
A recurring pattern: the human specifies a behavior vaguely, the AI implements a reasonable interpretation, and the result doesn't match what the human wanted. The human concludes the AI "got it wrong." But the AI can only implement what it was told — vague requirements produce valid but wrong implementations. Diagnosing this correctly is essential for improving the collaboration.
When output doesn't match expectation, first ask: "Did I specify that expectation clearly?" Before concluding the AI failed, check if you provided a specification it could have succeeded against.
✗ Pitfall
Starting new sessions without re-grounding the AI
Sessions that started with "can you continue from last time?" — without providing context — produced generic solutions that ignored project-specific constraints. The AI doesn't have memory between sessions. Every session is a fresh start for the model, even if the human is continuing from yesterday. The overhead of re-grounding is real, but the cost of skipping it is higher.
Never assume the AI knows the project history. Start every session by pointing to context files and summarizing the current state. This is not optional for complex systems work.
✗ Pitfall
Delegating architectural decisions without realizing it
Sometimes a question that feels like an implementation detail is actually an architecture decision. "Should we cache this on disk or in memory?" sounds tactical, but the answer determines the system's behavior on restart, crash recovery, and performance. If you say "you decide," you've delegated an architecture decision. Some of these are fine to delegate; others aren't. Knowing the difference requires understanding your own system well enough to recognize what's architectural.
Maintain a list of "decisions I own" — core architectural principles you've chosen. Before answering "you decide," ask whether this is a decision that should be on that list.
Surprises

Things That Weren't Expected

Findings that challenged assumptions about how human–AI collaboration on complex systems works.

Surprise
The collaboration generated its own meta-problem: context management
It was unexpected that managing the AI's context across sessions would become a first-class design problem — requiring its own artifacts (the Manifest, the Tier System, the skills files). This is a genuinely novel problem that didn't exist before AI collaboration became the development mode. It's not discussed in most AI tool tutorials, but it's the most important unsolved problem in long-term human–AI collaboration on complex projects.
Surprise
AI speed created a new failure mode: understanding lag
Traditional development is slow enough that the developer builds understanding as they build the system. AI-assisted development is fast enough that the human can be running code they don't understand within minutes. The "understanding lag" — the gap between system complexity and the human's mental model — is a new failure mode specific to AI-augmented development. It's not well-named in the literature yet, but it was the source of many debugging difficulties.
Surprise
The AI was better at explaining than teaching
Claude could explain anything when asked — but didn't proactively teach concepts that the human would need to understand to make good decisions later. The human's learning happened in response to questions, not from the AI volunteering to build the human's conceptual understanding. This matters for education: AI collaboration is not automatically pedagogically rich for the learner. Designing for learning requires explicit intention.
Surprise
The process revealed more about the domain than the technology
A significant fraction of the development work wasn't "working with AI" — it was understanding the faculty job well enough to specify it. The AI collaboration forced a level of explicit articulation that working manually wouldn't have required. In this sense, building the system generated tacit-to-explicit knowledge conversion about the faculty workload problem that has value independent of the tool.
