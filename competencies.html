Competencies

What Skills This Process Revealed

Analyzing the Focal development process as a competency lens, identifying the skills that enabled effective human–AI collaboration, the gaps that made it harder, and what that means for teaching.

Observed in Practice Teaching Implications Living Analysis
🔬
Methodology note: This analysis treats the Focal development process as a naturalistic case where the researcher is also the practitioner. Competency claims are grounded in observed behaviors across development sessions, not self-report alone. This section will be refined iteratively as the case continues to develop.
Competency Analysis

Skills Observed in the Collaboration

Each entry shows the competency, its observed status in this case, an example from the development process, and a note on teaching implications. Status tags reflect observation, not judgment.

Systems Thinking: Seeing Structures, Not Just Features
The ability to decompose a messy problem into a multi-layer architecture (WBS → Master → Sessions) and maintain design consistency across those layers. This was not AI-generated. It came from the practitioner's capacity to think in systems before implementation began.
Example in the case The three-layer architecture was designed before any code was written. When implementation revealed problems (e.g., the status overwrite bug), the fix came from understanding the systemic relationship between layers.
Teaching implication Case analysis of multi-layer systems before implementation, appropriate for CS intro and Software Design courses.
Requirement Articulation: Converting Tacit Knowledge to Explicit Specification
The ability to translate vague domain needs ("I don't know where my time goes") into concrete, testable requirements ("When I click Add to Work Sessions, a new entry should appear linked to the current task and project, with no manual fields to fill"). This is the primary bottleneck in human–AI system design.
Example in the case The Quick Add feature was specified precisely enough on first attempt that Claude implemented it correctly without revision cycles. Features where requirements were vague required 2–4 back-and-forth cycles.
Teaching implication Spec writing exercises where the specification is tested by AI implementation, appropriate for Software Engineering and Instructional Design courses.
Context Design: Designing for the AI's Cognitive Constraints
The practitioner recognized that the AI's context window is a finite resource and designed the Tier System to manage it. This is a meta-competency: understanding how the AI works and designing the collaboration structure around it. Most practitioners don't do this. They treat AI sessions as independent and wonder why quality degrades.
Example in the case The _MANIFEST.md Tier System and the skills files (encoded domain knowledge for recurring tasks) were both human-designed solutions to the AI's architectural constraints, significantly reducing session startup overhead.
Teaching implication Assign students to design a context management system for a multi-session AI project, in a new course type focused on AI collaboration infrastructure.
Iterative Refinement: Willingness to Redesign, Not Just Debug
When the monolithic script became unwieldy, the practitioner chose refactoring over patching. This required recognizing architectural debt early, and directing a non-trivial refactor before the problem became critical. The willingness to do this, rather than add features to a bad foundation, is a competency, not just a preference.
Example in the case The v0.1 → v0.2 refactor (monolith to modular package) was initiated by the practitioner, not triggered by a failure. The system worked; the structure was the problem.
Teaching implication Build a mandatory refactor milestone into studio and capstone projects, not as a punishment for bad code, but as a designed learning moment.
AI Output Verification
Knowing how to systematically verify that AI-generated code does what you think it does, not by running it once, but by understanding its logic, testing edge cases, and checking against requirements. This is distinct from general testing skills; it specifically addresses the "I'll trust it because the AI wrote it" bias.
Example in the case Certain behaviors that worked correctly under typical conditions were not systematically tested against edge cases until they surfaced in real use. This illustrates why AI-generated output benefits from deliberate verification beyond confirming that the common path works.
Teaching implication Structured code review rubric and verification protocol assignments, embedded in all AI-integrated courses, not just dedicated testing courses.
Managing Scope in AI-Speed Development
Because the AI generates features quickly, there is constant temptation to add "one more thing" before verifying what's already built. This produces large amounts of interdependent code that is hard to debug. Teaching humans to set and enforce session boundaries is an underappreciated skill that doesn't appear in traditional SE education.
Example in the case When development momentum is high and the AI generates features quickly, it is tempting to continue extending before verifying what is already working. The resulting interdependencies between untested additions can make later diagnosis harder to isolate.
Teaching implication Session planning with explicit scope limits, followed by retrospective analysis, appropriate for Project Management and Software Engineering courses.
Failure Mode Diagnosis: AI Error vs. Specification Failure
When AI-generated code is wrong, it's not always clear whether the AI made an error or whether the human's specification was ambiguous. Developing the diagnostic instinct to distinguish these two failure modes, and responding appropriately, is a specific competency that most practitioners lack.
Example in the case When something goes wrong in AI-generated code, the instinctive response is often to look at the AI's output first. Over time, the more productive habit is to examine the specification alongside the output. The failure frequently lives at the boundary between what was intended and what was actually communicated.
Teaching implication Failure attribution exercises using AI-generated code with planted bugs (some from AI errors, some from ambiguous specs), appropriate for debugging and QA courses.
API Literacy: Understanding External System Constraints
Understanding the constraints of APIs you're integrating with deeply enough to make good design decisions. Partial knowledge leads to designs that work in the happy path but fail at known API limitations. The AI knows APIs well in the abstract, but the practitioner needs to know the specific, current, real constraints of the system they're working with.
Example in the case API constraints that are understood at one point in development can fade into the background during later design conversations, especially across sessions. The pattern suggests that known constraints belong in persistent design documentation, not just in memory.
Teaching implication Read-the-docs first assignments before any AI-assisted implementation, appropriate for Web Dev and integration courses.
