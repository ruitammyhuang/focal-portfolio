Section 4 · Competencies

What Skills This Process Revealed

Analyzing the Focal development process as a competency lens — identifying the skills that enabled effective human–AI collaboration, and the gaps that made it harder. Both have direct implications for what we need to teach.

Demonstrated Strengths Revealed Gaps Curriculum Implications
🔬
Methodology note: This analysis treats the Focal development process as a naturalistic case — the researcher is also the practitioner. Competency claims are grounded in observed behaviors across development sessions, not self-report alone. This section will be refined iteratively as the case continues to develop.
4.1 · Demonstrated Strengths

Competencies That Enabled Effective Collaboration

Skills the practitioner brought that made human–AI collaboration on a complex system work well — and that are worth explicitly teaching.

Strong
Systems Thinking — Seeing Structures, Not Just Features
The ability to decompose a messy problem into a multi-layer architecture (WBS → Master → Sessions) and maintain design consistency across those layers. This was not AI-generated — it came from the practitioner's capacity to think in systems before implementation began.
Evidence in the case The three-layer architecture was designed before any code was written. When implementation revealed problems (e.g., the status overwrite bug), the fix came from understanding the systemic relationship between layers.
Strong
Requirement Articulation — Converting Tacit Knowledge to Explicit Specification
The ability to translate vague domain needs ("I don't know where my time goes") into concrete, testable requirements ("When I click Add to Work Sessions, a new entry should appear in my Work Sessions database linked to the current task and project, with no manual fields to fill"). This is non-trivial and is the primary bottleneck in human–AI system design.
Evidence in the case The Quick Add feature was specified precisely enough on first attempt that Claude implemented it correctly without revision cycles. Contrast with features where requirements were vague — they required 2–4 back-and-forth cycles.
Strong
Context Design — Designing for the AI's Cognitive Constraints
The practitioner recognized that the AI's context window is a finite resource and designed the Tier System to manage it. This is a meta-competency: understanding how the AI works and designing the collaboration structure around it. Most practitioners don't do this — they treat AI sessions as independent and wonder why quality degrades.
Evidence in the case The _MANIFEST.md Tier System and the skills files (encoded domain knowledge for recurring tasks) were both human-designed solutions to the AI's architectural constraints. They significantly reduced session startup overhead.
Strong
Iterative Refinement — Willingness to Redesign, Not Just Debug
When the monolithic script became unwieldy, the practitioner chose refactoring over patching. This required recognizing architectural debt early (before it became critical), making a case for the investment cost, and directing a non-trivial refactor. The willingness to do this — rather than add features to a bad foundation — is a competency.
Evidence in the case The v1 → v2 refactor (from monolith to modular package) was initiated by the practitioner, not triggered by a failure. The system worked; the structure was the problem.
4.2 · Revealed Gaps

Competencies This Process Showed We Need to Teach

Places where the collaboration was less effective than it could have been — gaps that aren't unique to this practitioner, but that AI-assisted development makes visible.

Gap
AI Output Verification→ Teach This
Knowing how to systematically verify that AI-generated code does what you think it does — not by running it once, but by understanding its logic, testing edge cases, and checking against requirements. This is distinct from general testing skills; it specifically addresses the "I'll trust it because the AI wrote it" bias.
Evidence in the case The status overwrite bug ran undetected for multiple sessions because the behavior was correct in the common case. Only when edge cases occurred was the error found. A structured verification protocol would have caught it earlier.
Gap
Managing Scope Creep in AI-Speed Development→ Teach This
Because the AI generates features quickly, there is constant temptation to add "one more thing" before verifying what's already built. This produces large amounts of interdependent code that is hard to debug because you don't know which part is broken. Teaching humans to set and enforce session boundaries is an underappreciated skill.
Evidence in the case Several sessions that started with one clear objective added 2–3 features before testing any of them. Debug sessions that followed were harder because the fault could be in any of the additions.
Gap
Distinguishing AI Knowledge Limitations from Human Specification Failures→ Teach This
When AI-generated code is wrong, it's not always clear whether the AI made an error or whether the human's specification was ambiguous. Developing the diagnostic instinct to distinguish these two failure modes — and responding appropriately to each — is a specific competency that most practitioners lack.
Evidence in the case Early bugs were often attributed to the AI when they were actually specification failures. Recognizing this improved the quality of requirements prompts in later sessions.
Mixed
API Literacy — Understanding External System Constraints
Understanding the constraints of APIs you're integrating with deeply enough to make good design decisions. Partial knowledge leads to designs that work in the happy path but fail at known API limitations. The AI knows APIs well in the abstract, but the practitioner needs to know the specific, current, real constraints of the system they're working with.
Evidence in the case Notion API's inability to filter by relation in queries was known but repeatedly forgotten during design conversations — leading to proposals that had to be revised once the constraint was re-stated.

What to Teach and When
