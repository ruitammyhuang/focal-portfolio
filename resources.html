Resource 1

Before You Build: Project Kickoff Checklist

What you need to have figured out before the first design conversation with AI — the things that cannot be delegated.

  • 1
    Define the problem in terms of user experience, not features

    Describe what your life looks like with the problem unsolved, and what it would feel like solved. "I want a dashboard" is a solution. "I never know which project is on fire until it's too late" is a problem the AI can help you design toward.

  • 2
    Enumerate your constraints explicitly

    What external systems must the solution integrate with? What APIs exist? What are you unwilling to change (e.g., existing Notion workspace structure)? What's your technical skill floor? The AI will design around what you tell it — unstated constraints become surprises later.

  • 3
    Decide what "success" looks like — specifically

    Name the behavior that will tell you the system is working. "I can see all overdue tasks across all projects in under 10 seconds, without opening Notion" is testable. "Better workload visibility" is not. You need both the scenario and the acceptance criterion.

  • 4
    Identify what you don't know about your own requirements

    Before the first conversation, write down 3–5 things you're uncertain about — workflow details you haven't fully thought through, edge cases you're unsure of. These become the agenda for early clarification dialogue.

  • 5
    Decide who owns each type of decision

    Establish upfront: the human owns problem definition, requirements, and final architecture calls. The AI proposes, implements, and flags issues. Confusion about this leads to the AI making design decisions the human didn't mean to delegate.

Resource 2

Session-Level Collaboration Protocol

How to structure each individual AI collaboration session for complex systems work — especially when the project spans many sessions over time.

💡
Core insight from Focal: Each AI session starts cold. The AI has no memory of prior sessions unless you design a context system. Designing that system is one of the most important — and most overlooked — tasks in long-term human–AI collaboration.
🗂️
Build a Tiered Context System
Designate a small set of "always read" files that give the AI enough context to work correctly — architecture overview, key IDs, core rules. Separate these from implementation files that only need reading when modifying. The Focal _MANIFEST.md Tier System is a working example.
🎯
Start Every Session With a Clear Objective
Before the AI reads any files, state what you want to accomplish in this session. This primes the AI to read selectively rather than exhaustively — and forces you to have a clear goal before you start.
✅
End Every Session With a Status Update
Before closing, update your context files with what changed. Which files were modified? What decisions were made? What's still open? This is the "commit message" for your human–AI collaboration — essential for session continuity.
🔍
Verify Before Trusting
AI-generated code should be understood, not just run. Before calling a session "done," test the behavior against your acceptance criteria on real data. Especially test edge cases the AI might not have known to consider.
Resource 3

Prompting Strategies for Complex System Design

Patterns that consistently produced better outputs in the Focal collaboration — and anti-patterns to avoid.

✓ Effective: Give the context, not just the task

Instead of: "Add a workload dashboard"

Say: "I want a page that shows, for each project in my Projects DB, how many work sessions I've logged this month and total hours. The goal is to see at a glance if I'm spending time proportional to what I think my priorities are."

The context tells the AI why, which enables it to make better micro-decisions while implementing.

✓ Effective: State constraints as constraints

Instead of: (leaving it implicit)

Say: "The Notion API does not support relation filters in queries — we'll need to filter client-side. Don't propose an approach that relies on filtering by relation in the API call."

AI models will generate solutions that violate constraints they don't know about. Make them explicit.

✗ Avoid: Asking for too much at once

Don't ask: "Build the full dashboard including workload view, focus tasks, and a quick-add panel all in this session"

Better: One coherent feature per session — plan to test before building the next. AI can generate a lot of code quickly; the bottleneck is your ability to understand and verify it.

✗ Avoid: Accepting without understanding

Don't do: Copy-paste AI-generated code and move on.

Better: Ask the AI to explain any section you don't understand before deploying. If you can't explain why the code does what it does, you won't be able to debug it when it fails — and it will fail.

⚠️
The speed trap: AI generates code much faster than humans traditionally write it. This creates pressure to move faster than your understanding allows. The gap between "AI wrote it" and "I understand it" is where most errors live — and where the most important learning happens.
Resource 4

The Iterative Design Loop for Complex Systems

A repeatable pattern for each design-build-reflect cycle — adapted from the Focal development experience.

A
Step 1
Ground the AI in current state
Point the AI to your context files. State what's working, what's not, and what the session objective is. This is not overhead — this is the work that prevents the AI from solving the wrong problem.
B
Step 2
Co-design before implementing
Before writing a line of code, discuss the approach. Ask the AI to propose 2–3 options and explain tradeoffs. Choose deliberately. This step is frequently skipped under time pressure — and is the source of most "we built the wrong thing" outcomes.
C
Step 3
Implement incrementally
Build one coherent piece. Get it working and understood before adding the next. For complex systems, "incremental" might mean: schema → API calls → logic → UI → integration test — in separate steps.
D
Step 4
Test against real data and real use
Run the feature against your actual system, not synthetic test cases. Real data surfaces edge cases the AI didn't anticipate. This is irreplaceable — the human must do this step, not the AI.
E
Step 5
Reflect and update context
After each cycle: What did you learn about the requirements? What design decisions were made and why? What's the system state now? Update your context files. This reflection converts experience into knowledge — and sets up the next session for success.
