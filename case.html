The Case

The Problem, the System, and Why It Was Complex

Before examining the collaboration, we need to understand what was being built and why it represented a genuinely non-trivial design challenge, not something a template or off-the-shelf tool could solve.

System Design Problem Context Case Rationale
The Problem

The Faculty Workload Tracking Problem

Academic faculty jobs are notoriously difficult to manage. The challenge is not any single area of work, but the sheer variety and unpredictability.

A typical faculty job has no clean boundaries. On any given day, the same person is a teacher designing course materials, a researcher managing a multi-author manuscript, a program administrator handling student inquiries, a committee member, a mentor, and a professional development learner, simultaneously and reactively.

The result: it is nearly impossible to know, objectively, where time actually goes. Calendar entries don't capture task-level effort. Todo lists don't connect to projects. Project boards don't roll up into a workload picture. The faculty member either over-engineers a system (and spends time maintaining it instead of working) or works without visibility into their own load distribution.

❌ Before Focal
  • Manual todo lists that don't connect to projects
  • No visibility into actual hours per area
  • Task completion not tracked across projects
  • Context-switching with no record of what was dropped
  • Workload decisions made on feel, not data
  • Reactive to whoever was loudest, not what was due
✅ After Focal
  • Per-project WBS databases auto-sync to a master tracker
  • Work sessions logged with task + project + type
  • Dashboard showing hours by project and work type
  • Focus view: overdue, due today, due this week (prioritized)
  • Status propagates automatically through the layers
  • Zero manual data entry overhead for tracking

The System

What Focal Is

Focal is a local Python/Flask web application that acts as an intelligent bridge between Notion databases and a unified workload view.

📋
Layer 1: Project WBS Databases
Each project has its own Work Breakdown Structure database in Notion, the place where tasks are created and edited by the user. One database per project, living on that project's hub page.
🗂️
Layer 2: Master WBS Tasks
A single implementation-tracking database fed by the Python sync tool. All project tasks appear here, enabling cross-project views, status tracking, and priority scoring. It is never edited directly. Changes always originate in Layer 1.
⏱️
Layer 3: Work Sessions
Time-logging entries auto-created when the user starts working on a task via the Focal UI. Each session captures task, project, work type, and duration, feeding the workload dashboard.
⚡
The key design insight: The automation runs one-directionally: from project WBS into master, never back. This prevents sync loops, protects formula-computed status fields, and keeps the user's editing mental model simple. The user edits in Layer 1; the system handles everything else.
The Focal UI (runs at localhost:8765)

Focal is a locally-run Flask web app with four views:

Main Hub: sync controls and a Quick Add form for creating tasks without opening Notion.

📌 Focus Tasks: a prioritized view of overdue, due-today, and due-this-week tasks across all projects.

📊 Workload Dashboard: hours logged by project and work type, showing actual vs. estimated workload distribution.

📐 System Design: the live design document for the system, maintained inside the app itself.

Complexity Rationale

Why This Was a Genuinely Complex Design Problem

Not all "let's build something with AI" projects are equally revealing. Here's what made this one substantive.

1
Complexity dimension
Multi-layer data architecture with real constraints
The system required three interlocking databases with specific sync directionality, formula-based status propagation, and idempotent sync logic. Getting one layer wrong broke the others. There was no safe "just try it and see" approach.
2
Complexity dimension
Evolving requirements that changed the architecture
The project started as a simple sync script and evolved into a modular Flask application with a dashboard. Each evolution required re-designing decisions made in prior iterations, not just adding features, but rethinking structure.
3
Complexity dimension
Integration with an external API (Notion) with quirks
Notion's API has well-documented limitations (e.g., relation filters not supported, formula fields read-only, rate limits). Every design decision had to account for what the API could and could not do, requiring the human to develop mental models of the external system.
4
Complexity dimension
The human was also the user, not an abstract requirements doc
Unlike a client project, the designer had rich tacit knowledge of the problem but had to learn to articulate requirements in ways the AI could act on. This surfaced important questions about the translation cost between human intent and machine-actionable specification.
5
Complexity dimension
Production system with real consequences
Unlike a tutorial project, Focal is the system the faculty member actually uses daily. Bad design decisions had real costs: broken sync, lost data, unusable interfaces. This created authentic stakes that shaped every decision.
Case Value

Why This Makes a Good Human–AI Collaboration Case

What distinguishes this from a generic "I used ChatGPT to build an app" story.

🔭
Long Time Horizon
The collaboration spans multiple months across multiple sessions, making it possible to study how human–AI collaboration evolves, how context is maintained, and how the human's approach changed over time.
📐
Full Design Lifecycle
The case covers requirements elicitation, architecture design, implementation, testing, refactoring, and ongoing maintenance. This spans the full software design lifecycle, not just code generation.
🪞
Reflexive Practitioner
The human designer is a CS Education and Instructional Design researcher, capable of naming what's happening pedagogically and methodologically while it's happening.
⚖️
Balanced Human + AI Roles
The human made all design decisions and owned all requirements. The AI generated code, surfaced options, and flagged problems. Neither was passive. The case shows authentic division of cognitive labor.
