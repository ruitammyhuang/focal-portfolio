The Process

How Focal Was Built: Phase by Phase

The development of Focal wasn't a straight line. It was an iterative, collaborative process with real design decisions, wrong turns, and deliberate pivots — all in conversation with Claude AI.

5 Major Phases Iterative Design Living Document
📖
How to read this section Each phase describes what the human brought to the collaboration, what the AI contributed, what design decisions were made, and what triggered the move to the next phase. The goal is to surface the cognitive division of labor — not just what was built, but who did what kind of thinking.
The Phases

Development Timeline

Click any phase to expand. Each phase is characterized by a distinct problem framing, tool state, and collaboration pattern.

Before any code existed, the work was conceptual: articulating a vague feeling of workload chaos into a tractable system design problem. This phase involved describing the faculty job structure to Claude, identifying what "tracking workload" actually meant, and designing the initial Notion database structure.

The initial setup was Notion-only: per-project pages with simple task lists. Claude helped design the page hierarchy, database schemas, and naming conventions. No automation yet.

Human's roleDomain expert — described work areas, project types, what "useful" would look like. Made all structural decisions.
AI's roleStructured the problem, proposed database schemas, surfaced tradeoffs between flat vs. relational Notion setups.
Key decisionPer-project WBS databases (not a single flat task list) — allowing project-level customization while still enabling a global view.
Key artifactInitial Notion workspace architecture: hub pages, WBS databases, Projects DB, Work Sessions DB.
What triggered the next phase Manual syncing between project WBS and any global view was untenable. As soon as there were 3+ projects, the cognitive overhead of keeping them consistent became the new problem.

The first version of Focal was a single Python script (~1,000 lines) that discovered all WBS databases in Notion by title pattern, pulled tasks from each, and created/updated corresponding entries in Master WBS Tasks. Claude wrote most of the code; the human defined the logic, approved the API strategy, and tested against real Notion data.

A Flask UI was added as a thin wrapper to make sync triggerable from a browser rather than the terminal. This was the first time the "Quick Add" concept appeared — the user wanted to create tasks without opening Notion.

Human's roleDefined sync rules (idempotency, one-directional), tested with real data, identified bugs, described edge cases from actual use.
AI's roleImplemented Notion API calls, wrote deduplication logic, built the Flask wrapper, debugged API quirks.
Key decisionOne-directional sync (Project WBS → Master only). Protects the Auto Status formula field in Master from being overwritten.
Key artifactnotion_wbs_sync.py — a ~1,000-line monolithic script with Flask routes, Notion client, and sync logic all in one file.
What triggered the next phase The monolithic script became hard to navigate and modify. Adding a new field or fixing a bug required understanding 1,000 lines of context. The human needed to describe changes to Claude, who would then search for the right location — inefficient and error-prone.

The monolith was refactored into a proper Python package — focal/ — with separation of concerns: API client, sync engine, route blueprints, templates. The human directed the architecture; Claude performed the refactor, maintaining behavior while restructuring.

This phase introduced formal documentation: a _MANIFEST.md Tier System to help Claude navigate the codebase in future sessions without reading all 3,000 lines. This was a meta-level insight: the collaboration was generating context management overhead that required its own designed solution.

Human's roleDirected the architecture split, defined the Tier System for context management, validated that behavior was preserved.
AI's roleExecuted the refactor, reorganized files, updated imports, built the Blueprint pattern for Flask routes.
Key decisionThe Tier System in _MANIFEST.md — a human-designed solution to the AI's context window limitations. Tier 1 files always safe to read; Tier 2 only when modifying; Tier 3 never unless asked.
Key artifactThe focal/ package: app.py, config.py, notion_client.py, sync_engine.py, tasks.py, routes/, templates/.
What triggered the next phase Syncing worked, but the user had no way to see the output. Tasks were in Notion but understanding "what's most urgent across everything" still required manually scanning each project. A focus view was needed.

Two new views were added to the Focal UI: a Focus Tasks view (overdue, due today, due this week — across all projects, auto-prioritized) and a Workload Dashboard (hours logged by project and work type, updated from Work Sessions data). Both required new Notion API queries and a caching layer to avoid hitting rate limits on every page load.

The Focus Tasks view also connected to Notion pages in Cowork — Claude could query it directly to answer "what should I work on today?" A scheduled task was set up to refresh the focus list daily.

Human's roleDefined what "focus" meant (urgency logic), specified which data the dashboard needed, tested the prioritization against actual workload perception.
AI's roleImplemented the Notion query logic, built the caching layer, designed the dashboard UI, created the scheduled task integration.
Key decisionCache focus tasks locally (not re-query Notion on every view) — necessary because the Notion API is slow and rate-limited for 200+ tasks.
Key artifactfocus.html, workload.html, focus-task-list-cache.json, dashboard_routes.py.
What triggered the next phase Daily use surfaced friction the design hadn't anticipated: the Master WBS schema felt awkward in practice, data entry was inconsistent across projects, and interface flows had accumulated small inefficiencies. A deeper restructuring was needed — not just new features.

Major restructuring of the Master WBS Tasks database — schema simplified, field organization redesigned to reduce redundancy and improve cross-project comparability. The data flow between Project WBS → Master WBS → Work Sessions was streamlined to eliminate unnecessary intermediate steps. Interface optimizations across the Focal UI reduced clicks for common actions and created more consistent visual hierarchy. Data entry was made more flexible and consistent through field defaults, option normalization, and validation rules applied across all project WBS databases.

This was a user-experience driven iteration, not a feature iteration — grounded entirely in friction encountered during daily use.

Human's roleIdentified friction from lived use, specified what felt wrong and why, directed the redesign priorities.
AI's roleProposed schema reorganization options, implemented DB restructure, rebuilt UI flows, normalized field options across all WBS databases.
Key decisionTreat UX friction as a first-class design signal — not a bug to patch but a signal that the architecture needed rethinking.
Key insightThe gap between "what seemed right in design" and "what works in daily practice" is a critical feedback loop that only the human-as-user can close.
What triggered the next phase System is stable and in daily use. The work now shifts to extracting transferable knowledge from the case — treating the development process itself as a research site.

The system is in active daily use and continues to evolve. The current focus is on: (1) treating the development process as a research site, (2) building this case portfolio as a structured artifact for CS Education and Instructional Design, and (3) refining design documents as the system matures.

New features are still being added — but the main intellectual work now is reflective and generative: extracting transferable knowledge from the case and making it useful for teaching and research.

Critical Moments

Turning Points in the Collaboration

Moments where the collaboration revealed something important about how human–AI system design actually works.

🧠
The Manifest Insight
When the codebase grew large enough that starting each session with "re-read all this code" was expensive, the human designed the Tier System — a structured way to tell the AI what's safe to read proactively vs. only when needed. This was a human designing for AI cognitive constraints, not the other way around.
🚨
The Status Overwrite Bug
An early sync implementation overwrote the Auto Status formula field on every sync run. This silently corrupted data. The fix — setting status only on CREATE, never on UPDATE — became a core design principle. The human's domain knowledge was essential; the AI had generated correct code that broke correct logic.
💬
Requirements That Changed After Implementation
Several times, a feature was implemented correctly but the human discovered the requirement was wrong once they saw it working. This is a well-known problem in software engineering — but AI-speed implementation made the feedback loop fast enough that it happened more frequently than in traditional development.
📐
Skills in Config Files
The system uses skills — structured prompt files — that encode domain knowledge (database IDs, sync rules, hub page patterns) so Claude can act correctly without re-deriving it each session. This blurred the line between documentation and automation: the skills are both human-readable and machine-actionable.
